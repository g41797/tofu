    Sockets 101

    But you said earlier that we don't need to know anything about sockets?

    tofu uses the same terms (like connect, etc.), you still need to understand the address scheme and the correct order of socket's creation, difference between client and server side. I asked Gemini to provide ABSOLUTE MINIMUM

    What is a Socket?

    A socket is a software endpoint. It allows two programs to communicate. Communication can be on the same computer. It can also be across a network.

    Stream-Oriented Sockets

    These sockets use a reliable, ordered, and connection-based protocol. TCP (Transmission Control Protocol) is the main example. Data is sent as a continuous stream of bytes. It guarantees that all data arrives correctly.

    Socket Families (Protocols)

    Sockets use different communication protocol families.

    1. TCP/IP Sockets

    This family is for network communication. It uses IP addresses to identify machines. TCP/IP is the foundation of the internet.

    2. Unix Domain Sockets (UDS)

    This family is for local communication only. It works only on the same computer. UDS does not use network hardware. It is often faster than TCP/IP for local processes.

    Socket Operations

    Sockets follow a set lifecycle for communication.

    1. Create

    The first step is to create the socket. This reserves system resources. It gives you a handle (file descriptor) for later use.

    2. Client Operation: Connect

    A client uses the connect() operation. It tries to link its socket to the server's address. If successful, the communication stream is open.

    3. Server Operations: Listen and Accept

    A server must wait for client connections.

    • Bind: The server attaches the socket to a specific local address.
    • Listen: The server waits for incoming connection requests.
    • Accept: When a client connects, the accept() call returns.
      • The accept() operation creates a second, new socket.
      • The original socket remains the listener socket. It keeps waiting for new clients.
      • The newly accepted socket is used for data transfer with the specific connected client.
      • This is why the server needs two sockets for one connection.

    4. Disconnect (Close)

    Communication ends when the socket is closed.

    • Graceful Close: One side signals it is done sending data. The other side receives the remaining data. This is a clean shutdown.
    • Non-Graceful Close: A sudden close, like a program crash. Data that is currently being sent may be lost.

    Addresses - TCP/IP

    TCP/IP addresses are a combination of an IP address and a port number.

    Server Address (Bind)

    The server binds to its local IP address. It uses a well-known, fixed port number (e.g., 22, 80, 443).

    Listening on a Specific Adapter A web server binds to the address 192.168.1.10:80. * 192.168.1.10 is the specific IP address of one network card (adapter). * The server will only accept connections arriving at that specific IP.

    Listening on All Adapters (Wildcard Address) To accept connections from any network card on the machine, the server uses a special address: 0.0.0.0. Example: A server binds to 0.0.0.0:8080. * 0.0.0.0 is the wildcard address. * This tells the operating system to listen on all available network adapters for connections on port 8080. This is very common for public servers.

    Client Address (Connect)

    The client uses the server's host name or IP address. It uses the server's fixed port number. The client's own socket is given a temporary ephemeral port. The operating system automatically assigns this temporary port. It is released when the connection closes.

    Example: A client connects to the web server using the address 192.168.1.10:80. The client's own socket might automatically be assigned the address 10.0.0.5:54321, where 54321 is the ephemeral port.

    Addresses - Unix Domain Sockets (UDS)

    UDS does not use IP addresses or port numbers.

    Path Used for Both

    UDS uses a file system path instead. The server must bind its socket to this path (e.g., /tmp/service.sock). The client must use this exact path to connect. This path acts as a local address for inter-process communication.

    Example: A server binds to the path /var/run/myservice.sock. A client connects using the exact same path: /var/run/myservice.sock.

    How the OS (Linux) Keeps Track of Sockets

    When your program creates a socket, the operating system must remember it and give the program a way to use it.

    File Descriptor (FD) = Socket ID / Handle

    • In Linux, every socket has a small number called a File Descriptor (FD).
    • This number works like a "handle" or "ID".
    • Your program uses this number to read from the socket, write to it, or close it.
    • Example: socket_fd = 5 → you write send(5, ...) or close(5).

    Important: FD is unique only inside one process

    • Process A can have a socket with FD = 3.
    • Process B can also have its own socket with FD = 3 (no problem!).
    • The same FD number can be used by many different programs at the same time.
    • The operating system knows the real socket because it looks at both:
      • the Process ID (PID) and
      • the File Descriptor number together.
    • So (PID + FD) is what is truly unique on the whole computer.

    Tuning Sockets (setsockopt)

    The setsockopt function allows a program to tune socket behavior.

    SO_REUSEADDR (Reuse the Same Port Quickly)

    When a TCP connection closes, the socket often goes into a state called TIME_WAIT. This state can last 1–4 minutes (depending on the system).

    While a socket is in TIME_WAIT: - The operating system normally does not allow a new program to bind (listen) on the exact same local address and port.

    This can be a problem when: - You want to restart your server quickly (after a crash or normal stop). - You are testing and starting/stopping the server many times.

    Note

    tofu uses SO_REUSEADDR for every listening socket

    SO_REUSEADDR fixes this problem: - When you turn on SO_REUSEADDR before calling bind(), - A new server can immediately use the same port again, even if old connections are still in TIME_WAIT.

    Important safe notes: - It is almost always safe to use SO_REUSEADDR on servers. - You should usually enable it when writing server programs that need to restart fast.

    SO_LINGER (Linger Option)

    Normally when you close a socket: - The close() function returns immediately. - If there is still unsent data, the operating system tries to send it in the background (this is called graceful close).

    The SO_LINGER option lets you change this default behavior.

    There are three possible settings:

    • Linger OFF (this is the default)
      • close() returns immediately.
      • The OS tries to send any remaining data in the background (graceful close).

    tofu Non-Graceful Close

    tofu uses THIS SETTING in order to achieve NON-GRACEFUL CLOSE

    • Linger ON with timeout = 0

      • close() returns immediately.
      • The connection is closed forcefully right away (hard close / reset).
      • All unsent data is thrown away.
      • This is the fastest way to close a socket when you don’t care about unsent data.

    • Linger ON with timeout > 0 (for example, 5 seconds)

      • close() will wait (block) up to the timeout value.
      • It waits for all remaining data to be sent and acknowledged.
      • If everything is sent before the timeout → graceful close.
      • If the timeout runs out → the socket is closed forcefully, unsent data is lost.

    Summary:

    • Default (linger off) → close fast, OS tries to be nice.
    • Linger on + timeout 0 → close fast and rude (reset).
    • Linger on + timeout > 0 → wait a little to be nice, but give up after timeout.
